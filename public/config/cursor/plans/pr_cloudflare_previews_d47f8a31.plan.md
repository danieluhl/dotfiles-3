# PR Cloudflare Preview Deployments

## Assumptions
- Scope is the `web` site only, not a separate per-PR `api` Worker.
- Preview builds should use the existing GitHub secrets/vars already used by deploys: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, `VITE_WORKOS_CLIENT_ID`, `VITE_WORKOS_API_HOSTNAME`, and `SERVER_URL`.
- The stable PR URL should be an aliased Cloudflare Worker preview URL such as `pr-123-web.<account>.workers.dev` format as generated by Cloudflare, not a custom domain.

## Existing Context
- [`package.json`](package.json) already has `deploy:web:preview` as `pnpm build:web && pnpm --filter web deploy:preview`.
- [`apps/web/package.json`](apps/web/package.json) already maps `deploy:preview` to `wrangler versions upload`.
- [`pnpm-workspace.yaml`](pnpm-workspace.yaml) pins `wrangler` at `^4.93.0`, which supports aliased preview URLs.
- [`apps/web/wrangler.jsonc`](apps/web/wrangler.jsonc) defines the `web` Worker and staging/production names.
- [`README.md`](README.md) already documents non-production branches using `npx wrangler versions upload`.
- [`github/workflows/deploy.yml`](.github/workflows/deploy.yml) deploys production/staging on branch pushes and should remain unchanged for mainline deploys.

Relevant existing command:

```json
"deploy:web:preview": "pnpm build:web && pnpm --filter web deploy:preview"
```

## Proposed Approach
Add a new PR-only GitHub Actions workflow, likely [`github/workflows/preview.yml`](.github/workflows/preview.yml), rather than mixing preview deployment into the existing branch deploy workflow. The workflow will:

1. Trigger on `pull_request` events for opened, synchronized, reopened, and ready-for-review PRs.
2. Skip draft PRs unless explicitly ready for review.
3. Install pnpm `10.20.0` and Node `25.3.0`, matching existing CI and deploy workflows.
4. Run `pnpm install --frozen-lockfile` and `pnpm check`.
5. Build `web` with `pnpm --filter web build`.
6. Upload a Cloudflare Worker preview version using the generated config:
   - `pnpm --filter web exec wrangler versions upload --config dist/server/wrangler.json --preview-alias "pr-${{ github.event.pull_request.number }}"`
7. Capture the preview URL from Wrangler output and post or update a PR comment with the URL.

## GitHub Permissions and Secrets
The preview workflow should request only what it needs:

- `contents: read` to check out the repo.
- `pull-requests: write` or `issues: write` to create/update the PR comment.

It will use the same Cloudflare secrets as [`github/workflows/deploy.yml`](.github/workflows/deploy.yml). If the repo blocks secrets on forked PRs, the workflow should only deploy for trusted PR contexts and let CI still run for forks.

## Cloudflare Behavior
This follows Cloudflare's documented Worker Preview URL flow:

- `wrangler versions upload` creates a Worker version and returns a preview URL.
- `--preview-alias` creates a stable human-readable alias for that version.
- Preview URLs are public when enabled; Cloudflare Access can be added later if previews need authentication.

Sources:
- https://developers.cloudflare.com/workers/configuration/previews/
- https://developers.cloudflare.com/workers/ci-cd/external-cicd/github-actions/index.md

## Verification
- Open or update a test PR and confirm the `Preview` workflow runs.
- Confirm the workflow posts one updated PR comment with the Cloudflare preview URL.
- Push another commit to the same PR and confirm the `pr-N` alias updates to the newest build.
- Confirm `Deploy` still only runs on `master` and `production` pushes.
- Confirm preview deployment does not require production secrets beyond the existing Cloudflare deploy token and build-time public/env vars.
