-## Authorize URL (`/authorize`)
-
-App-mode URL contract:
-
-```
+```text
 GET https://auth.anedot.com/authorize
-  ?app=<slug>
-  &return_to=<relative path>     # validated against app's returnToPolicy

-  &debug=1                       # optional
+  ?return_to=<absolute URL>
+  &auth_state=<signed token>
+  &state=<optional>
+  &debug=1
 ```
 
-`auth_state` is signed by the consuming app with **its own session secret**
-and verified by the auth app. The token's payload is
-`{ slug, returnTo, nonce, iat }`; the signature is HMAC-SHA256 over the
-base64url-encoded payload.
-
-### Outcomes (in order)
+`auth_state` is HMAC-SHA256 over a base64url payload:
 
-1. **Unknown / malformed app slug** → redirect to `/error?reason=unknown_app`.
-2. **`auth_state` fails verification** (bad signature, slug mismatch,
-   expired) → redirect to `/error?reason=invalid_auth_state`.
-3. **`return_to` violates the app's policy** →
-   `/error?reason=invalid_return_to`.
-4. **App's `authToken` cookie still valid** → silently redirect back to
-   `<origin><return_to>`.
-5. **App's `refreshToken` cookie present** → auth app silently refreshes via
-   Cognito and, on success, redirects back to `<origin><return_to>`.
-6. **Otherwise** → render the sign-in screen. After successful credentials,
-   MFA, force-change-password, or confirm-sign-up, the auth app writes the
-   three cookies and redirects to `<origin><return_to>`.
-
-`return_to` is always combined with the app's first `allowedOrigins` entry
-when the policy is `relative-only`, so the consuming app cannot smuggle a
-cross-origin destination through the parameter.
-

+```json
+{
+  "purpose": "authorize",

-## What a consuming app must do
-
-A new consumer must:
-
-1. **Run on TanStack Start** (or use a compatible session-cookie library) so
-   it can read the cookies the auth app writes.

-
-There is no need to call Cognito directly, exchange codes, or store refresh
-tokens. The auth app owns those pieces.
